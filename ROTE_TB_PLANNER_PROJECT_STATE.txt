SECTION 1 — WHAT THE APPLICATION IS
────────────────────────────────────────────────────────────────────────────────

The ROTE TB Planner is a self-contained desktop application for guilds playing
Star Wars: Galaxy of Heroes (SWGoH). It helps guild officers plan territory
battle (TB) phases — specifically the "Rise of the Empire" (ROTE) TB format.

Architecture: A single Python file (rote_planner_v6.py) that:
  1. Embeds a complete HTML/CSS/JavaScript single-page application (SPA) as a
     Python string (HTML_APP = r"""..."""), served over a local HTTP server.
  2. Runs a ThreadingHTTPServer on port 8080 for the web interface.
  3. Launches and manages a "comlink" Node.js proxy on port 3000, which
     provides authenticated access to the SWGoH game servers.
  4. Exposes a REST-like API at /api/* for the frontend to call.
  5. Is distributed as a single .exe via PyInstaller (--onefile).

The application is entirely local — no cloud services, no external APIs except
comlink (which communicates with EA's SWGoH game servers). The one exception
was an AI integration that has since been removed.

Build command:
  python -m PyInstaller --onefile --name ROTE_Planner rote_planner_v6.py

SECTION 2 — COMLINK PROXY: HOW THE GAME API WORKS
────────────────────────────────────────────────────────────────────────────────

WHAT COMLINK IS:
  Comlink is an open-source Node.js proxy that authenticates with EA's SWGoH
  game servers and exposes a local HTTP API. The app auto-detects comlink in
  several locations, can download it if missing, and auto-restarts it on crash.

COMLINK LOCATION SEARCH ORDER:
  1. .comlink/ subfolder beside the exe
  2. ./comlink (same directory)
  3. ~/comlink (home directory)
  4. Auto-downloaded if not found

COMLINK VERSION: v4.x (POST-only, all requests require enums:false)

COMLINK ENDPOINTS USED:
  POST http://localhost:3000/player
    Body: {"payload": {"allyCode": "123456789"}, "enums": false}
    OR:   {"payload": {"playerId": "base64-guid"}, "enums": false}
    Returns: player object with rosterUnit array

  POST http://localhost:3000/guild
    Body: {"payload": {"guildId": "guild-id-string"}, "enums": false}
    Returns: guild object with member list

  POST http://localhost:3000/data
    Body: {"payload": {"segment": N}, "enums": false}
    Segment numbers 1, 3, 6, 10, 11, 12 tried for unit definitions
    Returns: game data (unit definitions, ability data, etc.)

  POST http://localhost:3000/localization
    Body: {"payload": {"id": "Loc_ENG_US.txt", "unzip": true}, "enums": false}
    Pattern in localization: UNIT_MAGMATROOPER_NAME = "Magma Trooper"
    This is the intended way to get unit display names from comlink.

COMLINK v4 CONFIRMED FIELD NAMES (from scan_log.json analysis):
  Unit keys returned by /player in rosterUnit[]:
    - definitionId: "BASEUNIT:SEVEN_STAR" (strip suffix after ":" to get defId)
    - currentRarity: integer 1-7 (star level) — NOT "rarity"
    - currentTier: integer 1-13 (gear level) — NOT "gear"
    - currentLevel: integer (character level)
    - currentXp: integer
    - relic: object with {currentTier: N} where tier 1=not reliced, 3=R1...9=R7
      So relic_level = max(0, relic.currentTier - 2)
    - skill: array of {id: "skillname_DEFID", tier: N}
    - purchasedAbilityId: array of ability ID strings (zeta/omicron purchased)
    - equippedStatMod: array (present for characters, absent/empty for ships)
    - equipment: array
    - unitStat: CONFIRMED PRESENT BUT VALUE IS None/null IN BASIC ENDPOINT
      Speed stats are NOT available from the basic /player endpoint.
      Speed requires the stats endpoint or swgoh.gg private API.
    - combatType: NOT RETURNED by comlink v4 basic endpoint
      Must be inferred from defId or ship detection logic.

ZETA/OMICRON DETECTION (comlink v4):
  Zetas and omicrons are stored in purchasedAbilityId[].
  Each entry is an ability ID string.
  If the ability ID contains "omicron" (case-insensitive) → count as omicron.
  Otherwise → count as zeta.
  IMPORTANT: purchasedAbilityId was EMPTY for all members in last scan log.
  This may be because the basic endpoint doesn't return this data, or because
  the members hadn't purchased any. The high_relic_purchased field in scan
  log will show this on next scan.

SHIP DETECTION (comlink v4):
  combatType field is absent. Ships are detected by:
  Python side: _SHIP_DEFIDS set (103 entries) checked against defId
  JavaScript side: _SHIP_IDS set (~50 entries) in filterRoster()
  Also: characters have equippedStatMod (mods); ships typically don't.

ALLY CODE HANDLING (CRITICAL):
  Guild member ally codes from the /guild response come as base64 URL-safe
  encoded GUIDs, e.g., "fo8f8goIT2mjZKtRXZh8-w"
  These MUST NOT have hyphens stripped. Hyphens are significant characters.
  Only numeric ally codes (e.g., "659-388-776") should have hyphens stripped.
  Fix applied: check if raw.replace("-","").isdigit() before stripping.

SCAN FAILURES (fixed in current version):
  16 of 49 members failed in previous scans due to hyphen-stripping bug.
  All 49 now scan successfully (0 failures confirmed in latest scan log).

COMLINK RELIABILITY:
  - Background watchdog thread checks every 15 seconds if comlink is alive
  - Pipe drain threads prevent stdout/stderr buffer blocking crashes
  - Auto-restart on crash with exponential backoff
  - /api/comlink-health endpoint checks liveness and triggers restart

UNIT NAME RESOLUTION FROM COMLINK:
  The basic /player endpoint does NOT return character display names.
  nameKey field is MISSING from rosterUnit entries.
  The /localization endpoint (Loc_ENG_US.txt) contains all names in pattern:
    UNIT_{DEFID}_NAME = "Display Name"
  The /data endpoint with various segment numbers may return unit definitions
  with nameKey fields, but results vary by comlink version.
  Current fallback: comprehensive static UNIT_NAMES table in JS (270 chars).

FALLOFF SYSTEM:
  CM and Fleet completion rates drop off for later planets in each chain.
  Controls:
    - Base CM Rate (% of guild that completes each CM)
    - Base Fleet Rate (% that completes fleet)
    - CM Falloff (% decrease per zone deeper in chain)
    - Fleet Falloff
    - CM Mode: "% of members" or "exact count"
  Formula: effectiveRate = clamp(base - falloff * chainDepth, 0, 100)
  Individual planet overrides bypass falloff calculation

UNDEPLOY TABLE:
  Per-day adjustment for undeployed GP
  Mode: percentage of guild GP or flat GP amount
  Used by gpForDay() to calculate available GP each day

POINT CALCULATION (calcPlanetPts):
  total = cmPts + fleetPts + opsPts + gpDeploy + preloaded
  cmPts = planet.cms × round(activeMemberCount × cmRate) × planet.cmPts
  fleetPts = planet.fleets × round(activeMemberCount × fleetRate) × planet.fleetPts
  opsPts = filledPlatoons × planet.opsVal
  gpDeploy = manual override input
  preloaded = banked points from previous day

    the greedy future simulator (DS chain consumed all GP, LS got nothing).
    Fixed with proportional GP allocation in simulateFuture().

TAB 3: DAY-BY-DAY PLAN (id="dayplan", panel="panel-dayplan")
═══════════════════════════════════════════════════════

PURPOSE: Multi-algorithm optimizer to find the best 6-day deployment plan.

HEADER METRICS:
  - Total Est. Stars (synced with Guild Overview Est. Stars via _lastPlanStars)
  - Max possible stars (56 base + N bonus)
  - Bonus planets unlocked count
  - Operations platoons filled total

ALGORITHM SELECTOR (dropdown):
  Options:
    - "All Algorithms (Best Result)" [default]
    - "Genetic Algorithm"
    - "Simulated Annealing"
    - "Particle Swarm Optimization"
    - "Adaptive Moment Estimation (Adam)"
    - "Greedy Enumeration (Rule-Based)"

PROGRESS FILL BUTTON:
  The "Run Optimization" button itself fills with gold color as progress is made
  Status line below shows: algorithm name, % complete, best score so far
  When "All" selected: each algorithm gets 20% of the progress bar

GENOME ENCODING:
  Flat array of 18 integers: [d0_ds, d0_mx, d0_ls, d1_ds, d1_mx, d1_ls, ... d5_*]
  6 days × 3 chains = 18 decisions
  Gene values: 0=preload, 1=target 1★, 2=target 2★, 3=target 3★

FITNESS FUNCTION (evalGenome):
  Simulates all 6 days given a genome.
  For each day:
    - Compute mission points for each active chain planet
    - Compute GP needed for each chain's target star tier
    - PROPORTIONAL GP ALLOCATION: if total need > budget, each chain
      gets share = (chain_need / total_need) × budget. No chain starved.
    - Apply decisions: preload banks points (capped below 1-star threshold)
      commit advances to next planet if ≥1 star achieved
    - BONUS PLANETS: after main chains, spare GP goes to unlocked bonus planets
      bonusUnlocked() returns true only if planet's SM count meets threshold
      Bonus planet stars are included in the fitness score
  Returns: integer total stars across all 6 days + bonus planets

ALL 5 ALGORITHMS:

  1. Greedy Enumeration (runGreedy):
     Wraps existing runOptimizer() as a genome.
     runOptimizer() enumerates all 64 daily combinations (4 options × 3 chains)
     Uses lookahead (simulateFuture) to score each combination.
     Provides the seed genome for all other algorithms.

  2. Simulated Annealing (runSA):
     ITERS=10,000, T0=120, Tf=0.05, geometric cooling
     Starts from greedyGenome()
     Perturbation: 2-4 genes when T > T0*0.5, else 1-2 genes
     Accepts worse solutions with probability exp(delta/T)
     Yields every 400 iterations for UI thread

  3. Particle Swarm Optimization (runPSO):
     SWARM=60 particles, ITERS=250
     W=0.72 (inertia), C1=1.49 (cognitive), C2=1.49 (social)
     Particles in continuous [0,3] space, quantized to int for evaluation
     3 particles seeded near greedyGenome() for warm start
     Velocity clamped to [-2, 2]

  4. Genetic Algorithm (runGA):
     POP=200, GENS=180, ELITE=8 (elitism), MUT=0.12 (mutation rate)
     Uniform crossover between tournament-selected parents
     Tournament size k=6
     Occasional whole-day block mutation (5% chance)
     First generation seeded with greedyGenome()

  5. Adam AME (runAdam):
     AGENTS=40, ITERS=350, LR=0.45, B1=0.9, B2=0.999, EPS=1e-8
     Central-difference gradient: evaluates ±0.5 perturbations per gene
     Gradient ascent (maximizing stars)
     2 agents seeded near greedyGenome(), rest random

  When "All" selected: Greedy → SA → PSO → GA → Adam run sequentially
  Best genome by score is used for final display.

ALGORITHM COMPARISON HEADER:
  When multiple algorithms run, a gold-highlighted comparison box shows
  each algorithm's best score. Winner is highlighted.

DAY DISPLAY (renderOptPlan):
  For each day 1-6:
    - Planet name and chain alignment
    - Status: "Preloading" (banked pts shown) or N-star commit
    - GP available / GP used
    - Stars earned that day
    - Bonus planet results (spare GP applied)
  Bonus planet notes shown below each day's chain cards.

AUTO-RUN:
  When navigating to Day-by-Day tab for the first time (empty output),
  startOptimization() fires automatically.
  Subsequent visits do not auto-run.

  - The _lastPlanStars sync ensures Guild Overview matches, but is invalidated
    on any settings change (ops toggle, stats change, falloff change).
  - buildAIPrompt() function still exists in code (dead code — no longer called)
    but doesn't harm anything. Safe to remove.

  Body: {allyCode: "string"}

  Returns: guild object from comlink or {error: "..."}

  Body: {allyCode: "base64-or-numeric"}

    - For numeric: POST to comlink /player with {allyCode: numeric}

    - Parse rosterUnit[] into simplified format
    - Store in _guild_rosters[raw_allycode]
    - Write to scan_log.json

STARTUP FLOW:
  1. User runs ROTE_Planner.exe
  2. Python unpacks embedded comlink binary to temp directory
  3. Python starts comlink on port 3000 with pipe drain threads
  4. Python starts HTTP server on port 8080
  5. Browser opens to http://localhost:8080
  6. HTML/JS app loads; calls /api/status to check comlink
  7. Watchdog thread starts (checks comlink every 15 seconds)

GUILD IMPORT FLOW:
  1. User enters ally code (their own or any guild member's)
  2. JS calls /api/guild-by-allycode with ally code
  3. Python strips hyphens (numeric only), sends to comlink /player
  4. Gets playerId from player response, then sends to comlink /guild
  5. Guild object returned: {guild: {name, gp, profile: [{playerId, allyCode, ...}]}}
  6. JS displays guild name, member count, GP
  7. Member ally codes stored for roster scanning

ROSTER SCAN FLOW:
  1. User clicks "Scan Rosters"
  2. JS iterates all guild member ally codes
  3. For each member: POST /api/roster with allyCode
  4. Python determines if GUID or numeric, calls comlink /player
  5. Response: {rosterUnit: [{definitionId, currentRarity, currentTier, ...}]}
  6. Python parses each unit:
     - defId = definitionId.split(":")[0]
     - stars = currentRarity
     - gear = currentTier  
     - relic = max(0, relic.currentTier - 2) where relic.currentTier 1=off, 3=R1
     - combatType = 2 if defId in _SHIP_DEFIDS else 1
     - speed = 0 (unitStat is null)
     - zetas/omicrons from purchasedAbilityId[] (may be empty)
  7. Simplified roster stored in _guild_rosters[allyCode]
  8. Debug info written to scan_log.json
  9. After all members: updateDefaultsFromRosterScan() adjusts planet rates
  10. populateMemberDropdown() updates Guides and Roster dropdowns

OPTIMIZATION FLOW:
  1. User configures Planet Planner (CM rates, fleet rates, ops, SMs, GP)
  2. User selects algorithm(s) and clicks "Run Optimization"
  3. startOptimization() runs selected algorithm(s) sequentially
  4. Each algorithm calls evalGenome() for fitness, yields to UI thread
  5. evalGenome() uses:
     - gpForDay(d): guildGP × participationRate × (1 - undepRate)
     - missionOnlyPts(pid): cm + fleet + ops points from pState
     - bonusUnlocked(id): checks pState[unlockedBy].smCount ≥ threshold
  6. Best genome selected by highest star count
  7. simulateGenomeDetailed() runs best genome for display output
  8. renderOptPlan() shows day-by-day breakdown
  9. _lastPlanStars = totalStars (syncs to Guild Overview Est. Stars)
  10. calcSummary() updates s-stars element

GUIDES ROSTER CHECK FLOW:
  1. User navigates to Guides tab
  2. User selects a planet → mission (e.g., Mustafar → CM 1)
  3. User selects a guild member from the member dropdown
  4. renderGuideMission() renders squad cards
  5. For each squad, checkRosterForSquad() called:

     b. For each member name in squad.members[]:

     c. Checks: unit.rarity === 7 AND unit.relic >= planet.relic
     d. Builds results[] with {name, ok, reason} per unit
  6. Squad header shows: ✓ Ready or ✗ N missing (with full tooltip)
  7. Member names in squad body individually colored ✓ green / ✗ red

SECTION 10 — PLANET/TERRITORY GAME MECHANICS
────────────────────────────────────────────────────────────────────────────────

ROTE TB STRUCTURE:
  6 days of play, 3 chains of planets
  Dark Side chain: Mustafar → Geonosis → Dathomir → Med Station →
                   Malachor → Death Star
  Mixed chain:     Corellia → Felucia → Tatooine → Kessel → Vandor → Hoth
  Light Side chain: Coruscant → Bracca → Kashyyyk → Lothal → Kafrene → Scarif
  Bonus planets:   Mandalore, Zeffo (unlocked by SM completions)

  Each day: one planet active per chain. Completing ≥1 star advances to next.
  Preloading: deploying GP but not reaching 1-star carries points to next day.
  Maximum stars: 56 (18 planets × 3 stars) + 6 (2 bonus × 3 stars) = 62 max

STAR THRESHOLDS (from swgoh.wiki, per planet):
  Each planet has 3 thresholds (total points needed for 1★/2★/3★).
  Thresholds vary by planet's zone level.
  Values are total guild-wide points (CM + Fleet + Ops + deployed GP).

POINT SOURCES:
  Combat Missions (CMs): each CM × number of completions × points per player
    "Completion" = a guild member finished that CM that day
    Points per player scales with zone level (200k to 1.15M per CM per player)
  Fleet Missions: similar to CMs but with fleet-specific point values
  Operations Platoons: each filled platoon adds a fixed point value
    Up to 6 platoons per planet (buttons 1-6 in Planet Planner)
    Point values: 10M (Z1) to 86.49M (Z6) per platoon
  Deployment: guild members deploy GP from their characters/ships
    Total deployable GP = guild GP × participation rate × (1 - undeployed %)
    GP deploys to the territory where characters are placed

SPECIAL MISSIONS:
  Some planets have one special mission (SM) requiring specific character squads
  SM completions count toward bonus planet unlock thresholds:
    Tatooine Krayt Dragon SM: 25 completions → unlocks Mandalore
    Bracca SM: 30 completions → unlocks Zeffo
  SM counts are manually entered in the Planet Planner (not auto-calculated)

RELIC REQUIREMENTS:
  Each zone has a minimum relic level for effective CM participation:
    Zone 1 (R5+), Zone 2 (R6+), Zone 3 (R7+), Zone 4 (R7+), 
    Zone 5 (R7+), Zone 6 (R8+) — approximate, varies by mission type
